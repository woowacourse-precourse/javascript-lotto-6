  - 그래서 챕터를 나눠서 `피드백, 목적` -> `코드 설명` 순으로 작성해서 후반부만 보고도 코드를 이해할 수 있도록 하였다.

### 기능 목록을 재검토한다
- 구현 시작 전 기능 목록을 작성할 때 클래스, 함수별로 작성하기 보다는 큰 틀로 작성하기 위해 노력하였다
- 필요한 기능과 예외를 먼저 작성하고, 구현을 진행하면서 어떤 기능을 구현해야 하는지 계속 확인하였다.

### 기능 목록을 업데이트한다
- 기능 목록을 보면서 구현하다가 부족한 부분이나 수정하고 싶은 부분이 생기면 추가적으로 작성해주었다.
- 완벽한 설계, 완벽한 코드는 없다고 생각하기 때문에 나에게 이어 최선의 코드를 작성하는 것을 목표로 계속 업데이트 하며 구현하였다.

### 값을 하드 코딩하지 않는다
- 상수로 값을 표현해서 해당 상수가 어떤 역할을 하는지 바로 드러날 수 있게 해주었다.
  ```js
  export const RANK = Object.freeze({
    first: { match: 6, reward: 2_000_000_000 },
    bonus: { match: 100, reward: 30_000_000 },
    second: { match: 5, reward: 1_500_000 },
    third: { match: 4, reward: 50_000 },
    fourth: { match: 3, reward: 5000 },
  });
  ```

- 상수로 표현하지 않은 문자나 값들이 있는데, 수학적인 계산을 위한 값(퍼센트를 구하기 위해서 100을 곱하는 경우)이나 문장부호(쉼표, 공백)은 처리하지 않았다.
  - 해당 상수의 역할이 명확하게 있고 해당 값이 바뀌면 모두 바뀌는 경우에만 상수화를 해야 한다고 생각한다.
  - 공백, 쉼표와 같은 경우는 어떤 부분에는 정말 출력을 위해서만 사용될 수도, 중요한 표시가 될 수도 있는데 같은 모양이라고 해서 하나로 통일하면 안될 것이라 생각했다.
  ```js
   #getSplitWinningNumbers(winningNumbers) {
      return winningNumbers.split(',').map((number) => number.trim());
   }
  ```

### 구현 순서도 코딩 컨벤션이다
- 클래스를 필드 -> 생성자 -> 메서드 순으로 작성하였다.
- 메서드들의 순서도 코드를 이해하는데 많은 영향을 끼칠 것이라고 생각해서 특정 메서드 -> 특정 메서드 내에서 호출되는 메서드 순으로 작성하였다.

### 한 함수가 한 가지 기능만 담당하게 한다
- 한 함수가 여러가지 일을 하고 있다면 해당 로직을 분리해서 다른 메서드로 작성하였다.
- 그래서 다른 기능이 필요하다면 그 기능을 구현한 다른 메서드를 호출하는 방식으로 사용할 수 있도록 하였다.

### 함수가 한 가지 기능을 하는지 확인하는 기준을 세운다
- 내가 생각한 한 가지 기능이란 해당 함수 내에 구현된 기능이 한가지인가? 였다.
- 다른 함수를 호출하는 것은 한가지 기능이라기 보단 해당 기능을 구현하기 위해서 해줘야 하는 처리로 보았고, 해당 처리한 값을 사용해서 딱 한가지의 일만 하고 있는가를 한가지 기능으로 보았다.

### JavaScript에서 객체를 만드는 다양한 방법을 이해하고 사용한다
- Class를 사용하는 것이 익숙하지 않아서 클래스에 대해 많이 공부하며 구현을 진행하였다.

### 테스트를 작성하고 이유에 대해 본인의 경험을 토대로 정리해본다
- 직접 실행해보아야지만 결과를 알 수 있다고 생각했는데 테스트를 작성하면서 그게 아니라는 것을 알게 되었다.
  - 어떤 케이스에 대해 여러번 검사하고 싶을 때 계속 실행해보는 대신 테스트 코드를 작성하면 test.each를 통해 여러가지 값을 한번에 확인할 수도 있고, 
    해당 테스트를 통해 어떤 값이 나와야 하는지 스스로 돌아볼 수도 있었으며 구현상의 부족한 점도 알아챌 수 있었다.


- 작성한 로직이 제대로 작성하는지 확인할 뿐만 아니라 더 좋은 코드를 작성할 수 있게 도와주는 역할도 한다.
  - 테스트를 작성하면서 테스트를 작성하기 어려운 상황에 부딪히면 어떤 이유때문에 테스트가 어려운지, 
    현재 테스트가 어려운 이유는 설계상에 어떤 문제가 있는 것은 아닌지 돌아보는 기회도 되주었다. 다양한 접근 방식으로 코드를 볼 수 있어서 좋았던 것 같다.

### 처음부터 큰 단위의 테스트를 만들지 않는다
- 가장 작은 단위의 테스트부터 작성하면서 각 함수가 제대로 작동하는지 확인하였다. 
- 처음부터 통합 테스트를 작성해버리면 오류가 났을 경우 어디가 문제인지 확인할 수 없기 때문에 핵심 기능부터 차근차근 테스트를 진행하였다.

## 📢 추가된 요구 사항

- 함수(또는 메서드)의 길이가 15라인을 넘어가지 않도록 구현한다.
  - 메서드가 하나의 일만 하도록 전부 분리해서 작성하였다. 그래서 메서드의 이름을 보면 어떤 일을 수행하는지 바로 확인할 수 있게 하였다.
  - eslint를 사용해서 `'max-lines-per-function': ['error', { max: 15 }]` 최대 길이르 15로 정해주었다.


- else를 지양한다.
  - else를 사용하면 예상치 못한 값들이 모두 else로 처리될 수 있기 때문에 if를 사용해서 정확하게 언제 적용되는지 나타내주었다.
  - 또한 Early Return을 사용해서 특정 상황에 바로 반환되게 해서 복잡성을 줄이며 간단하게 작성하였다.


- 도메인 로직에 단위 테스트를 구현해야 한다. 단, UI(Console.readLineAsync, Console.print) 로직에 대한 단위 테스트는 제외한다.
  - view(UI 로직)을 제외한 model, controller의 메서드에 대한 테스트를 작성하였다. 


- 사용자가 잘못된 값을 입력할 경우 throw문을 사용해 예외를 발생시킨다. 그런 다음, "[ERROR]"로 시작하는 에러 메시지를 출력하고 해당 부분부터 입력을 다시 받는다.
  - 다시 입력받아야하기 때문에 try-catch문을 사용해서 에러를 발생시키고, 다시 입력받을 수 있게 하였다.
  - 입력 -> 검증 -> 잘못된 입력인 경우 재입력 -> 검증 -> ... 


- Random 값 추출은 `Random.pickUniqueNumbersInRange()`를 활용한다.
  - 이전 주차에는 `pickNumberInRange()`를 사용했었는데 이번에는 `pickUniqueNumbersInRange()`을 사용해 중복되지 않게 여러개의 값을 선택하였다.


- 제공된 Lotto 클래스를 활용해 구현해야 한다.
  - Lotto 클래스의 위치를 model로 이동하였을 뿐 그대로 사용해서 구현하였다.
  - #numbers 필드를 사용해 로또의 번호를 저장하고, 로또의 순위를 결정하는 역할을 맡고있다.


### 이전 주차 요구 사항 적용
- indent(인덴트, 들여쓰기) depth를 3이 넘지 않도록 구현한다. 2까지만 허용한다.

## 🚀 목표
### ⭐️ 클래스(객체)를 분리하기
- 클래스를 분리하기 전 요구사항에 나온 도메인 로직과 UI 로직에 대해서 알아보았다. 그래서 내 나름대로 해당 로직에 대한 기준을 정해보았다.
  - `도메인 로직` : 소프트웨어로 해결하고자 하는 문제 영역 (= 핵심 로직)
  - `UI 로직` : 사용자와 소통하는 창구 역할 
- 그래서 입출력 로직을 따로 분리해서 작성하고, 게임의 핵심적인 부분은 도메인 로직에 작성하면 되겠다고 결정하고 구조를 설계하였다.


- 그 과정에서 model, view, controller로 역할을 분리해 설계하는 `mvc 패턴`이 도메인과 UI 로직을 분리하는데 좋을 것이라고 판단해서 적용하게 되었다.
  - 또한 이렇게 분리헤서 설계하게 되면 도메인 로직과 UI 로직을 수정하고 구현할 때 서로에게 큰 영향이 가지 않을 것이라 생각해 채택하게 되었다. 
- mvc 패턴을 처음 적용해보는 것이기 때문에 부족할 수 있지만 나름대로의 역할과 기준을 세워서 각 클래스가 제대로 작동할 수 있게 노력해보았다.

### ⭐️ 도메인 로직에 대한 단위 테스트를 작성하기
- `단위 테스트` : 소스 코드의 특정 모듈이 의도대로 정확히 동작하는지 검증하는 절차 -> 모든 함수와 메서드에 대한 테스트 작성
- 요구사항에서 UI 로직에 대한 단위 테스트는 제외한다고 해서, model과 controller의 메서드들에 대해 테스트를 작성하였다.
  - 좋은 테스트 스위트는 유지비용 문제가 있기에 코드에서 가장 중요한 부분을 대상으로 해야한다고 했는데 그 중요한 부분이 도메인 로직이 아닐까싶다.
- 모든 메서드들에 대한 테스트를 작성하였고, 각 메서드가 성공, 실패하는 테스트를 모두 작성하기 위해 노력하였다.
- 테스트 커버리지가 현재 100% 나오게 작성하였다. 하지만 테스트 커버리지는 그저 지표이기 때문에 수치보다는 효과적인 테스트를 작성하였는지에 목표를 두고 작성하였다.

## 주제 

> **로또 게임** 🎱 : 입력된 금액만큼 로또를 구매하고, 당첨 번호, 보너스 번호와 비교해서 당첨 결과와 수익률을 계산하는 게임 

---
## 프로그램 흐름

1. 로또 구입 금액을 입력받는다.
2. 금액만금 로또를 구매한다.
3. 로또를 구매하면 각 로또마다 중복되지 않는 6자리의 번호가 생성된다.
4. 구매한 로또의 번호를 출력한다.
5. 당첨 번호를 입력받는다.
6. 보너스 번호를 입력받는다.
7. 당첨, 보너스 번호와 각각의 로또 번호를 비교해서 등수를 구한다.
8. 등수에 맞는 수익금을 계산한다.
9. 전체 수익에 대한 수익률을 계산한다.
10. 등수와 수익률을 출력한다. 
11. 게임을 종료한다.

- getter 함수들은 작성하지 않음 (설명하지 않아도 어떤 기능인지 확인 가능하기 때문!)

### Model
#### 📍 Lotto.js
> 로또 번호를 저장하고, 해당 번호를 당첨, 보너스 번호와 비교해 순위를 구하는 클래스 (당첨번호도 로또에서 관리)

- `validate` : 로또 번호를 검증하는 기능 
- `sortNumbers` : 로또 번호를 오름차순으로 정렬하는 기능
- `countMatchingNumbers` : 다른 로또 번호와 비교해 일치하는 번호의 개수를 세는 기능 
- `hasBonusNumber`: 로또 번호 중 보너스 번호와 일치하는 번호가 있는지 확인하는 기능 
- `getMatchingCount` : 일치하는 번호의 수와 보너스 번호 포함 여부를 확인해 보너스 순위에 대한 처리 후 매칭 카운트 반환하는 기능

  - 다른 메서드들을 호출하는 메서드로 startGame 메서드를 보면 게임이 어떻게 진행되는지 확인 가능
- `getLottoBundle` : 로또 구매 금액을 입력받아서, 로또 번들 인스턴스를 반환하는 기능 (올바르지 않은 값 입력시 재입력받음)
- `getWinningLotto` : 당첨 번호를 입력받아서, 당첨 번호에 대한 로또 인스턴스를 반환하는 기능 (올바르지 않은 값 입력시 재입력받음)
- `getBonusNumber` : 보너스 번호를 입력받아서, 보너스 번호 인스턴스를 반환하는 기능 (올바르지 않은 값 입력시 재입력받음)
